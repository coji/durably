# アンビエントエージェントの概念

## 目的

このドキュメントでは、Durably のコアランタイム上に構築される拡張として、アンビエントエージェントの具体的なプロダクト像を描きます。

これはランタイム仕様ではありません。以下の3点を明確にすることが目的です。

- アンビエントエージェントとは一般に何か
- どの責務がエージェントレイヤーに属するか
- どのようなアプリケーションが自然な適用先になるか

## 作業定義

アンビエントエージェントとは、継続中の人間の業務や業務オブジェクトに時間をまたいで張り付き続ける、永続的でイベント駆動のワーカーです。

その主たる姿はチャット UI ではありません。

定義的な性質は連続性にあります。

- 環境の変化を観測する
- 必要なときに起動する
- 下書き、提案、アラート、アクションを生成する
- 安全に中断・再開し、人間へ作業を返せる

ユーザーから見ると、「質問すると答える AI」よりも、「見えないところで仕事を持ち続ける担当者」に近い存在です。

## 何が「アンビエント」なのか

アンビエントであるとは、通常は以下の5つの意味を含みます。

1. プロンプト駆動ではなくイベント駆動で動く
2. 単発リクエストではなく長寿命のセッションに結び付いている
3. 同じタスクに対して時間をまたいで複数回の run を実行できる
4. 常に対話を要求せず、有用なときだけ表に現れる
5. リロード、再起動、リトライ、フェイルオーバーをまたいでも連続性を維持する

そのため、アンビエントエージェントは通常、以下の要素を中心に構成されます。

- 永続的なセッション
- 再開可能な実行
- 追記専用のイベントストリーム
- 承認や割り込みのポイント
- 明示的な監査可能性

## コアモデル

プロダクト設計の観点では、以下の抽象モデルが有用です。

- **Session**: 継続中のタスクにおける連続性の境界
- **Run**: そのセッションに対して発火した1回の実行スライス
- **Step**: run の中の再開可能な1単位の作業
- **Event**: ユーザー可視または運用上の状態変化を表す永続的な記録
- **Artifact**: ユーザーまたはシステムが利用する永続的な成果物
- **Approval**: 不可逆な操作の前に人間が判断するポイント

このモデルにおいて:

- セッションは通常、業務オブジェクトや作業スレッドに紐付く
- run は webhook、スケジュール、閾値超過、ユーザー操作といったトリガーで生成される
- artifact はドラフト、スコア、サマリ、レコメンデーション、更新のような具体物を指す

## アンビエントエージェントではないもの

あらゆるエージェントやワークフローがアンビエントであるわけではありません。

通常、以下はアンビエントエージェントに該当しません。

- 1回限りの同期チャット完了
- 継続状態を持たない単発バッチジョブ
- 答えたら消えるリクエスト-レスポンス型のアシスタント
- 連続性、リプレイ、人間への引き継ぎを必要としないワークフロー

これらのユースケースも Durably のコアランタイムには適合しますが、アンビエントエージェントレイヤーまでは必要としません。

## なぜ Durably に適しているのか

Durably がアンビエントエージェントの下地として妥当なのは、難しい部分が単なるモデル呼び出しにとどまらないからです。

本当に難しいのはランタイムとしての連続性です。

- イベントで起床すること
- 安全に実行権を取得すること
- ワーカー障害をまたいで継続すること
- チェックポイントから再開すること
- ユーザー可視の進捗を保持すること
- 再接続やリフレッシュ後に状態を回復すること

そのため、アンビエントレイヤーは以下の機構の上に自然に築かれます。

- リースされた run
- 再開可能なステップ実行
- 永続的なチェックポイント
- 永続的なイベントストリーム

## アプリケーションパターン

最も自然なアプリケーションは、チャット中心ではなくオブジェクト中心になります。

UI の主キーは「会話」ではありません。通常は業務オブジェクトです。

- CRM のセグメント、キャンペーン、アカウント
- 候補者、求人、採用パイプライン
- プルリクエスト、リポジトリ、エンジニアリングチーム

エージェントはそのオブジェクトを時間をまたいで追い続け、その周辺で成果物を生み出します。

## 例1: CRM / 顧客ライフサイクルエージェント

### セッションの形

セッションは顧客アカウント、セグメント、キャンペーンサイクル、分析スレッドに紐付きます。

### 典型的なトリガー

- 新しい購買データや行動データが到着する
- セグメントが大きく変化する
- キャンペーン結果が更新される
- 月次計画サイクルが始まる
- 人間が追加の提案を求める

### 典型的な成果物

- 更新されたセグメント仮説
- 異常検知アラート
- 注力すべき顧客群の提案
- キャンペーンアイデアのドラフト
- 営業担当向けレポートのドラフト

### なぜアンビエントなのか

このエージェントは常にチャットする必要がありません。

価値は、顧客行動を継続的に追いながら、次に取るべきアクションを静かに更新し続ける点にあります。人間は必要なときにレビューし、実行すればよいのです。

## 例2: HR / ATS 伴走エージェント

### セッションの形

セッションは候補者、求人、面接ループ、採用案件に紐付きます。

### 典型的なトリガー

- 候補者が進行または停滞する
- 面接メモやトランスクリプトが追加される
- リクルーターが候補者ページを開く
- 次のアクションがないまま期限が近づく
- オファー作成が始まる

### 典型的な成果物

- 面接要約
- スクリーニングルールの提案
- 情報不足アラート
- リクルーターコメントのドラフト
- オファーや候補者連絡文面のドラフト

### なぜアンビエントなのか

採用は長寿命で、承認が多く、運用上もセンシティブなプロセスです。

有用なのは、案件を静かに追跡し、次の成果物を準備しておき、リクルーターや採用マネージャーの判断が必要な場面だけを前面に出すようなエージェントです。

## 例3: 開発生産性エージェント

### セッションの形

セッションはプルリクエスト、リポジトリ、チーム、エンジニアリング施策に紐付きます。

### 典型的なトリガー

- webhook でプルリクエストイベントが到着する
- レビュー待ち時間が閾値を超える
- 定期的な週次分析が始まる
- リリースやデプロイ状態が変化する
- 人間が傾向の説明を求める

### 典型的な成果物

- サイクルタイムの説明
- レビュー停滞アラート
- 週次サマリ
- ボトルネックの推定分析
- レトロスペクティブのドラフト

### なぜアンビエントなのか

開発者は通常、プロセスについて一日中アシスタントと会話したいわけではありません。

求められているのは、適切なタイミングで適切なシグナルやドラフトが現れることです。その裏では、リトライやバックフィルをまたいでも壊れない永続的な分析が動いている必要があります。

## これらの領域に共通するプロダクト特性

上記の例はいずれも同じ構造を共有しています。

- プロダクト内に既に主たる業務オブジェクトが存在する
- そのオブジェクトは時間とともに変化する
- その変化がトリガーを生む
- エージェントはそのオブジェクトの周辺で成果物を生成する
- 重要な操作の最終判断者は人間のままである

そのため、メンタルモデルとしては「チャットボット」よりも「見えない AI」のほうがふさわしいです。

エージェントは独立した会話画面として導入するより、既存のプロダクトフローに埋め込むほうが適切なことが多いです。

## アンビエントレイヤーの境界

アンビエントレイヤーが担うべき責務は以下の通りです。

- セッションライフサイクル
- run とセッションの関連付け
- 永続的なイベントの保存とリプレイ
- 高速復旧のためのスナップショットポリシー
- 承認および割り込みのフック

一方で、より上位のインテリジェンス関連をすべて定義する必要はありません。以下の要素はプラグイン可能なままに留められます。

- プランナー戦略
- モデル選択
- 検索やメモリのポリシー
- ドメイン固有のツール
- UI 表現の詳細

## 設計ヒューリスティック

以下の条件の大半を満たすなら、そのユースケースはアンビエントエージェントレイヤーへの適合度が高いです。

- タスクが数時間から数週間にまたがる
- 同じタスクに何度も再入できる
- 複数のトリガーが同じタスクを起こしうる
- 永続的な進捗や監査履歴が求められる
- ユーザーが明示的に依頼する前に作業を準備しておきたい
- 最終アクションには人間のレビューが依然として有益である

これらの性質がなければ、より単純なジョブランタイムで十分なことが多いです。
