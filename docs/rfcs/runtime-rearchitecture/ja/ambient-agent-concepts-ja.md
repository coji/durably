# 設計: アンビエントエージェントの概念

## 目標

本ドキュメントは、Durably のコアランタイム上に構築される拡張として、アンビエントエージェントの具体的なプロダクト像を描く。

これはランタイム仕様ではない。目的は次の3点を明確にすることにある:

- 一般にアンビエントエージェントとは何か
- どの責務がエージェントレイヤーに属するか
- どのようなアプリケーションが自然な適用先となるか

## 作業定義

アンビエントエージェントとは、継続中の人間の業務や業務オブジェクトに時間をまたいで張り付き続ける、永続的でイベント駆動のワーカーを指す。

その主たる姿はチャット UI ではない。

定義的な性質は連続性にある:

- 環境の変化を観測する
- 必要なときに起動する
- 下書き、提案、アラート、アクションを生成する
- 安全に中断・再開し、人間へ作業を返せる

ユーザーから見ると、「質問すると答える AI」よりも、「見えないところで仕事を持ち続ける担当者」に近い。

## 何が「アンビエント」なのか

アンビエントであるとは、通常は次の5つの意味を含む:

1. プロンプト駆動ではなくイベント駆動で動く。
2. 単発リクエストではなく長寿命のセッションに結び付いている。
3. 同じタスクに対して時間をまたいで複数回の run を実行できる。
4. 常に対話を要求せず、有用なときだけ表に現れる。
5. リロード、再起動、リトライ、フェイルオーバーをまたいでも連続性を維持しなければならない。

そのため、アンビエントエージェントは通常、次の要素を中心に構成される:

- 永続的なセッション
- 再開可能な実行
- append-only のイベントストリーム
- 承認や割り込みのポイント
- 明示的な監査可能性

## コアモデル

プロダクト設計の観点では、次の抽象モデルが有用となる:

- `Session`: 継続中のタスクにおける連続性の境界
- `Run`: その session に対して発火した1回の実行スライス
- `Step`: run の中の再開可能な1単位の作業
- `Event`: ユーザー可視または運用上の状態変化を表す永続的な記録
- `Artifact`: ユーザーまたはシステムが利用する永続的な成果物
- `Approval`: 不可逆な操作の前に人間が判断するポイント

このモデルにおいて:

- session は通常、業務オブジェクトや作業スレッドに紐付く
- run は webhook、schedule、閾値超過、ユーザー操作といった trigger によって生成される
- artifact は draft、score、summary、recommendation、update のような具体物を指す

## アンビエントエージェントではないもの

あらゆる agent や workflow がアンビエントであるわけではない。

通常、以下はアンビエントエージェントに該当しない:

- 1回限りの同期 chat completion
- 継続状態を持たない単発 batch job
- 答えたら消える request-response 型の assistant
- 連続性、リプレイや人間への引き継ぎを必要としない workflow

これらのユースケースも Durably のコアランタイムには適合しうるが、アンビエントエージェントレイヤーまでは必要としない。

## なぜ Durably に適しているのか

Durably がアンビエントエージェントの下地として妥当なのは、難しい部分が単なる model invocation にとどまらないからだ。

本当に難しいのはランタイムとしての連続性にある:

- event で起床すること
- 安全に実行権を取得すること
- worker 障害をまたいで継続すること
- checkpoint から再開すること
- ユーザー可視の進捗を保持すること
- reconnect や refresh 後に状態を回復すること

したがって、アンビエントレイヤーは次の機構の上に自然に築かれる:

- lease された run
- 再開可能な step 実行
- 永続的なチェックポイント
- 永続的なイベントストリーム

## アプリケーションパターン

最も自然なアプリケーションは、chat 中心ではなく object 中心になる。

UI の主キーは「会話」ではない。通常は業務オブジェクトだ:

- CRM の segment、campaign、account
- candidate、requisition、採用パイプライン
- pull request、repository、エンジニアリングチーム

エージェントはそのオブジェクトを時間をまたいで追い続け、その周辺で成果物を生み出す。

## 例1: CRM / 顧客ライフサイクルエージェント

### Session の形

session は customer account、segment、campaign cycle、analysis thread に紐付く。

### 典型的な Trigger

- 新しい購買データや行動データが到着する
- segment が大きく変化する
- campaign 結果が更新される
- 月次計画サイクルが始まる
- 人間が追加の提案を求める

### 典型的な Artifact

- 更新されたセグメント仮説
- 異常検知アラート
- 注力すべき顧客群の提案
- campaign アイデアの draft
- 営業担当向けレポートの draft

### なぜ Ambient なのか

このエージェントは、常にチャットする必要がない。

価値は、顧客行動を継続的に追いながら、次に取るべきアクションを静かに更新し続ける点にある。人間は必要なときにレビューし、実行すればよい。

## 例2: HR / ATS 伴走エージェント

### Session の形

session は candidate、requisition、interview loop、採用案件に紐付く。

### 典型的な Trigger

- candidate が進行または停滞する
- 面接メモや transcript が追加される
- recruiter が candidate ページを開く
- 次アクションがないまま期限が近づく
- offer 作成が始まる

### 典型的な Artifact

- 面接要約
- screening rule の提案
- 情報不足アラート
- recruiter コメントの draft
- offer や候補者連絡文面の draft

### なぜ Ambient なのか

採用は長寿命で、承認が多く、運用上もセンシティブなプロセスだ。

有用なのは、案件を静かに追跡し、次の成果物を準備しておき、recruiter や hiring manager の判断が必要な場面だけを前面に出すようなエージェントになる。

## 例3: 開発生産性エージェント

### Session の形

session は pull request、repository、team、エンジニアリング施策に紐付く。

### 典型的な Trigger

- webhook で pull request event が到着する
- review latency が閾値を超える
- 定期的な週次分析が始まる
- release や deployment 状態が変化する
- 人間が傾向の説明を求める

### 典型的な Artifact

- cycle time の説明
- review 停滞アラート
- 週次 summary
- ボトルネックの推定分析
- retrospective の draft

### なぜ Ambient なのか

開発者は通常、プロセスについて一日中 assistant と会話したいわけではない。

求められているのは、適切なタイミングで適切な signal や draft が現れることだ。その裏では、retry や backfill をまたいでも壊れない永続的な分析が動いている必要がある。

## これらの領域に共通するプロダクト特性

上記の例はいずれも同じ構造を共有している:

- プロダクト内に既に主たる業務オブジェクトが存在する
- そのオブジェクトは時間とともに変化する
- その変化が trigger を生む
- agent はそのオブジェクトの周辺で artifact を生成する
- 重要な操作の最終判断者は人間のままだ

このため、メンタルモデルとしては「chatbot」よりも「見えない AI」の方がふさわしい。

agent は独立した会話画面として導入するより、既存のプロダクトフローに埋め込む方が適切なことが多い。

## アンビエントレイヤーの境界

アンビエントレイヤーが担うべき責務は次のとおり:

- session lifecycle
- run と session の関連付け
- 永続的なイベントの保存とリプレイ
- 高速復旧のための snapshot policy
- 承認および割り込みの hook

一方で、より上位の intelligence concern を全て定義する必要はない。

次の要素は pluggable なままに留めてよい:

- planner strategy
- model choice
- retrieval や memory の policy
- domain-specific tools
- UI 表現の詳細

## 設計ヒューリスティック

次の条件の大半を満たすなら、そのユースケースはアンビエントエージェントレイヤーへの適合度が高い:

- タスクが数時間から数週間にまたがる
- 同じタスクに何度も再入できる
- 複数の trigger が同じタスクを起こしうる
- 永続的な進捗や監査履歴が求められる
- ユーザーが明示的に依頼する前に作業を準備しておきたい
- 最終アクションには依然として人間のレビューが有益だ

これらの性質がなければ、より単純な job runtime で十分なことが多い。
